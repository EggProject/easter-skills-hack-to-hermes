# 11 — Sub-Agent Delegation Map (Phase 5)

## Goal

Partition Phase 5 (Tasks B / C / D / E + Script #3) into independent sub-agent workstreams, each owning one or two major deliverables. Each workstream runs in its own worktree (no cross-worktree writes) and produces a PR that the parent coordinator reviews. The workstreams are sequenced by dependency but can OVERLAP once their pre-conditions are met.

## Pre-conditions (what every workstream assumes)

- `plans/00..12` is committed on `main` (HITL gate passed).
- `pyproject.toml`, `.pre-commit-config.yaml`, `uv venv` are set up on the worker's worktree base.
- The `hermes_home` and `hermes_checkout` fixtures are in `tests/conftest.py`.
- The `tools/check_bilingual.py`, `tools/check_line_count.py`, `tools/check_migration_note.py` hooks are installed.

## Workstream dependencies (DAG)

```
00..13 (committed on main)
   |
   +---> B-plugin ----+
   |                  |
   +---> C-script-1 --+
   |                  |
   +---> D-script-2 --+--> F-meta
   |       |          |     (final index update + per-file budget assertion)
   |       v          |
   |    G-report ------+
   |   (shares D-script-2's enabled-detection module)
   |
   +---> E-skill -----+
```

B / C / D / E are independent (no cross-deliverable imports). G-report depends on D-script-2 because both consume the same enabled-detection module (config/toggle + profile + platform; honor `platforms:` / conditional exclusions). F depends on all five (it asserts the per-file line-count budget across the whole plan set and updates 00-index.md with the final state).

## Sub-agent prompt template (per workstream)

```
You are the Phase 5 sub-agent for workstream <ID>. Read the plan files
<PLAN_FILES> in this worktree before writing any code. Load the pre-skills
<PRE_SKILLS>. Create the branch <BRANCH> from main.

Deliver in this order (TDD):
  1. Write the failing test(s) per the plan's TDD list.
  2. Make them pass.
  3. Refactor.
  4. Run `uv run pytest --cov=hermes_skill_creator_plugin --cov-branch
     --cov-fail-under=100` — must exit 0.
  5. Run `pre-commit run --all-files` — must exit 0.
  6. Commit per the granularity in 10-toolchain-and-conventions.md.
  7. Open a PR with the standard template. Title: "Phase 5 / <ID>: <summary>".

Hard rules:
  - NEVER write to ~/.hermes/hermes-agent. The hermes_home and hermes_checkout
    fixtures redirect writes to tmp_path.
  - NEVER use os.environ.pop on HERMES_SESSION in the parent process. Use the
    hermes_subprocess_env() helper.
  - ALL console messages MUST be bilingual (en/hu on a single line).
  - ALL --help output MUST be two-section (English / magyar) with mirrored content.
  - Do NOT hand-edit MIGRATION*.md; they are generated by --emit-migration-note.
  - For G-report (Script #3) ONLY: the script is READ-ONLY. Assert zero bytes
    written to HERMES_HOME during every test. Any write = test failure.

When done, return a summary covering: branch name, commit SHAs, test count,
coverage %, pre-commit status, PR URL, any deviations from the plan, and any
open questions for the parent.
```

## Pre-skill loadouts (per workstream)

| Workstream | Pre-skills to invoke before writing code |
| --- | --- |
| B-plugin | `update-config` (for any settings.json tweak), then `code-review` (post-write, before PR) |
| C-script-1 | `update-config`, `code-review` (post-write), `verify` (run `--check` against fixture) |
| D-script-2 | `update-config`, `code-review` (post-write), `verify` (run against fixture) |
| E-skill | `update-config`, `code-review` (post-write), `verify` (run the eval pipeline end-to-end) |
| F-meta | `code-review` (final pass on the whole plan set) |
| G-report | `update-config`, `code-review` (post-write), `verify` (run against fixture; confirm zero writes) |

`update-config` is for any harness / settings.json changes (e.g. adding a new allowed permission). `code-review` is mandatory before the PR opens — it catches the high-confidence reuse / simplification / efficiency cleanups at the `--comment` level. `verify` is for the workstreams that touch a script or a CLI: it confirms the change runs in the real (fixture) environment, not just in the test suite.

If two workstreams race to land the same module, the second one to land rebases onto the first and re-runs its tests. The pre-commit hook `check_bilingual` is the same lint for all five.

## PR review expectations (parent coordinator)

- Every PR is reviewed against the AC IDs in its commit messages AND against the per-file plan's TDD list.
- A PR that lands without `test_*` for a new function is rejected (red-flagged in `code-review`).
- A PR that lands with `os.environ.pop("HERMES_SESSION", None)` anywhere in the parent process is rejected (use `hermes_subprocess_env()`).
- A PR that touches `~/.hermes/hermes-agent` (caught by the `real_hermes_agent_sentinel` decorator) is rejected.
- A PR that places the migrated skill-creator under `src/hermes_skill_creator_plugin/` (i.e. embeds the standalone skill inside the plugin package) is rejected per Brief §5.4 / §6.D.6 — the skill MUST live at the worktree-root `skills/skill-creator/` and ship / install flat into `~/.hermes/skills/skill-creator/`.
- A G-report PR that writes to `HERMES_HOME` during tests or in `--help` / dry-run paths is rejected (the script is strictly read-only).

## Post-Phase-5 verification (Phase 6)

A separate sub-agent (workstream G-verify) runs the full suite on `main` after all five PRs are merged:
- `uv run pytest` — full test suite, 100% coverage enforced.
- `pre-commit run --all-files` — all hooks pass.
- `uv run hermes-skill-creator-patch --check --target <fixture-checkout> --task-e-redirect` — exits 0.
- `uv run hermes-skill-creator-profiles --apply` against `HERMES_HOME=tmp_path` — exits 0; per-profile JSON report produced.
- `uv run python -m hermes_skill_creator_plugin.install --yes` with `HERMES_HOME=tmp_path` — exits 0; skill is installed at `tmp_path/hermes-home/skills/skill-creator/` (flat install path, not plugin-bundled).
- `uv run hermes-skill-creator-report --profile hermes --sort tokens` with `HERMES_HOME=tmp_path` — exits 0; produces the sortable report; assert zero bytes written under `tmp_path/`.
- A static check that `~/.hermes/hermes-agent` is byte-identical to its pre-Phase-5 state (sha256 of `agent/skill_utils.py`).
- A static check that `src/hermes_skill_creator_plugin/` does NOT contain a `skills/` subdirectory (the migrated skill must be standalone at the worktree root).

Workstream G-verify is NOT in the Phase 5 DAG above; it runs as Phase 6 (separate sub-agent invocation, see plan #15 in the task list).
